# ADR-001: Adopting LangGraph StateGraph for Context-Aware Routing and Orchestration

## Context

The AI Retail Decision Copilot was initially designed as a sequential execution chain where all user requests passed through the same linear pipeline. Requests for quantitative calculations (e.g., top products, monthly sales trends) and qualitative document searches (e.g., return policies, store SOPs) went through identical processing blocks.

This linear pipeline approach suffered from:
1. **Inefficient Latency:** Qualitative document retrieval and context construction were executed even when a query could be answered using structured data alone.
2. **Context Pollution:** Merging unrelated document snippets with quantitative table analysis diluted the LLM prompt, occasionally leading to hallucinated trends or loss of mathematical accuracy.
3. **API Cost Inefficiency:** Running high-token document context through the LLM for every single interaction significantly inflated token usage and operating costs.

To achieve clean, resilient, and fast execution paths, the application required an orchestration framework capable of analyzing user intent and dynamically routing queries to specific analytical or document-retrieval pipelines.

## Decision

We chose to migrate from a sequential pipeline to a state-machine based directed graph architecture using **LangGraph StateGraph**. 

The implemented design organizes the application flow into discrete execution nodes:
1. **`intent_router`:** Classifies user query intent into `analytics` (structured data calculations), `rag` (unstructured document search), or `both`.
2. **`analytics_node`:** Invokes pandas-based analytical logic (e.g., top/bottom aggregations, trend growth rates, and Z-score anomaly calculations) on loaded dataframes.
3. **`rag_node`:** Ingests document questions, retrieves relevant chunks from the local FAISS index, and compiles them into a structured context string.
4. **`synthesiser`:** Evaluates the outputs of preceding nodes, reconciles quantitative and qualitative findings, and outputs the final response, complete with page and source document citations.

State is defined globally using a Pydantic-based `CopilotState` schema, enforcing data validation and structured boundaries between graph steps.

## Consequences

### Positive

* **Execution Efficiency:** Quantitative queries skip vector searches completely, resulting in faster responses and lower token cost.
* **Separation of Concerns:** Isolated nodes make the codebase highly modular. Developers can modify the RAG embedding setup or dataframe aggregations without breaking downstream LLM logic.
* **Resilience:** If the local FAISS vector store is uninitialized (e.g., no PDF uploaded), the graph handles the exception cleanly by routing to `analytics` or outputting a graceful fallback, rather than crashing the execution.
* **Context Isolation:** The synthesizer only receives context that is relevant to the classified path, drastically reducing LLM confusion.

### Negative

* **System Complexity:** Adopting LangGraph introduces additional boilerplate code, state management structures, and learning curves for maintainers.
* **Routing Latency:** Every user question requires an initial LLM call to classify the routing intent, which adds a minor latency overhead (approx. 100–300ms) before processing begins.

### Trade-offs

We chose to accept the slight overhead of the classification step (offset by using rapid LLM endpoints like `gemini-1.5-flash` or `gpt-4o-mini`) to achieve much cleaner context management, improved reasoning accuracy, and reduced token footprints in the final synthesis node.

## Alternatives Considered

* **Single LLM Prompt with Tool Calling:** We considered letting the LLM select and run tools dynamically using OpenAI/Gemini tool calling. This was rejected because tool-calling agents are less deterministic, harder to test, and can enter loops, whereas LangGraph gives developers full control over the execution graph structure.
* **Rule/Regex-Based Router:** We evaluated writing standard string parsing rules (e.g., regex checks for "sales", "average", "SOP"). This was rejected because business queries are highly expressive and conversational, meaning semantic routing via a lightweight LLM is far more robust and accurate.
